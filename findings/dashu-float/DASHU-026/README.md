# DASHU-026: exp_m1 panics across the negative range-reduction boundary in debug and fuzz builds

Status: confirmed on the locked baseline. Confidence: high. Classification: `panic`.

Contract: `backend_conformance`. Owner: `backend`. Masked by adapter: `true`.

Latest release check: Reproduces with dashu-float 0.5.0 and current Dashu master 40f465b62e5d8f4198efc43871e3ce601d03dc93 when debug assertions are enabled; the same 2,064-case sweep has no panics in a normal release build. No matching upstream issue or fix was found when verified on 2026-07-20.

## Summary

A generated sweep of 129 consecutive finite f64 values around the point where x/ln(2) reaches isize::MIN, eight precisions, and both Up and Down modes finds 1,035 exp_m1 panics in a debug build. The minimized case uses bits 0xc3d62e42fefa39ef, precision 2, and Up rounding. Dashu panics from its integer allocation guard with `out of memory`; the same sweep reports zero panics in a normal release build.

## Impact

Correctness and availability depend on the compilation profile for finite public inputs. Debug builds panic, and libFuzzer's abort-on-panic profile terminates the worker at the first case, preventing the rest of a campaign unless the range is subprocess-isolated. OpenDP's adapter preclassifies sufficiently negative primitive expm1 inputs and masks this raw backend path.

## Tested baseline

`dashu 0.5.0`, `dashu-float 0.5.0`, `dashu-int 0.5.0`, `dashu-ratio 0.5.0`, `malachite 0.9.2`, `rug 1.30.0`.

The repository-level `findings/verification.json` records the most recent automated replay. A separate latest-upstream check is required before filing if the status table does not say it was tested.

## Deduplication rationale

Separate from DASHU-023's non-panicking directed saturation after the range quotient no longer fits isize. DASHU-026 occurs in the adjacent finite range-reduction boundary region and manifests as a profile-dependent integer allocation panic before a value is returned.

## Reporting note

This is a direct provider probe generated by the raw FBig boundary sweep. The reproducer deliberately checks both compilation profiles because the observed behavior differs.
