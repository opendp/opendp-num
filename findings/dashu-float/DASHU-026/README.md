# DASHU-026: debug assertion in `round_fract` materializes `B^exponent_gap` (exp_m1 boundary panic + allocation growth)

Status: confirmed on the locked baseline. Confidence: high. Classification: `panic` + `resource-exhaustion` (one defect, two manifestations).

Contract: `backend_conformance`. Owner: `backend`. Masked by adapter: `true`.

Latest release check: Reproduces with dashu-float 0.5.0 and current Dashu master 40f465b62e5d8f4198efc43871e3ce601d03dc93 **when debug assertions are enabled**; the same probes are clean in a normal release build. No matching upstream issue or fix was found when verified on 2026-07-20.

## Summary

The generic rounding helper `Round::round_fract` (`float/src/round.rs:108`) validates its `fract` argument with a `debug_assert!(fract.unsigned_abs() < UBig::from_word(B).pow(precision))`. When an aligned subtraction produces a sparse sticky remainder (`add.rs:466`/`:367` set `low = (significand.signum(), ediff)`), that pair reaches `round_fract` (`add.rs:306`) as `fract = ±1`, `precision = ediff` — the exponent gap. The assertion then constructs `B^ediff`, allocating memory proportional to the gap. `FBig::exp_m1` reaches this via `exp(x) - 1` for large-magnitude finite `x`.

Two manifestations of the same line:

- **Allocation growth** (`resource-exhaustion`): `exp_m1(±1e8)`, precision 2, `Up` → 20,422,236 peak live heap bytes in debug vs. 788 in release.
- **Boundary panic** (`panic`): the 2,064-case negative-boundary sweep panics 1,035 times in debug (`out of memory`) and 0 times in release; minimized case bits `0xc3d62e42fefa39ef`, precision 2, `Up`.

Release is unaffected because the `debug_assert!` is compiled out and the actual comparison short-circuits on `log2_bounds` before reaching the `.pow(precision)` at `round.rs:126`.

## Impact

A compact finite input at tiny precision drives debug/libFuzzer memory into the tens of MB to GB, terminating debug services and fuzz workers; abort-on-panic profiles stall at the first boundary case. OpenDP's adapter preclassifies sufficiently large/negative primitive `expm1` inputs and masks this raw backend path.

`dashu 0.5.0`, `dashu-float 0.5.0`, `dashu-int 0.5.0`, `dashu-ratio 0.5.0`, `malachite 0.9.2`, `rug 1.30.0`.

## Reproduce

Run from the repository root:

```bash
cargo run --example reproduce_dashu_026   # 2,064-case boundary sweep
cargo run --example reproduce_dashu_027   # ±1e8 allocation probe
```

Both reproducers are retained because they exercise genuinely distinct observable symptoms (boundary OOM-panic vs. 20 MB growth) of the same root cause.

## Evidence

Reproduced directly against the library API; see the commands above. Root cause verified by source read of `float/src/round.rs:108,121-126` and `float/src/add.rs:306,367,466` at commit `40f465b`.

## Deduplication rationale

DASHU-026 and the former DASHU-027 are the same defect — a single debug-only assertion in generic rounding — and are merged. Distinct from DASHU-023's non-panicking directed saturation after the range quotient no longer fits `isize`: that is a range-reduction correctness error, whereas this is a generic-rounding allocation/assertion problem.
